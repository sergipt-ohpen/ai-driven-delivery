# 📜 Repository Management Policy | AI-Driven Delivery

**1\. Objective**  
To establish clear guidelines for the management of project repositories within the IA4Devs program, balancing students' professional visibility with the protection of sensitive information.

**2\. Repository classification**

**2.1 Public repositories**  
Repositories may be public when they meet the following conditions:

-   The project is presentable as part of the student's professional portfolio
    
-   It does not contain sensitive or confidential information
    
-   It includes adequate documentation (README) explaining the purpose and workings of the project
    

**2.2 Private repositories**  
Keeping repositories private is recommended in the following cases:

-   The project contains credentials, tokens, or access keys
    
-   Real APIs or services with sensitive data are used
    
-   There is internal information or information not intended for public distribution
    
-   The project is not ready to be shared externally
    

**3\. Mandatory best practices**  
For all repositories, especially public ones, the following practices must be followed:

-   Do not include .env files in the repository
    
-   Do not expose API keys, tokens, passwords, or credentials
    
-   Do not include real user data or personal information
    
-   Use .env.example files to document required variables
    
-   Use fictitious or test values in shared configurations
    
-   Include a README with clear installation and usage instructions
    

**4\. Responsibility**  
Each student is responsible for the content they publish in their repositories.  
IA4Devs is not liable for the exposure of sensitive information resulting from improper use of these practices.

**5\. Visibility approach**  
Publishing repositories as public projects is part of students' professional development, facilitating portfolio building and exposure within the tech ecosystem.

It is recommended to consciously evaluate which projects to share publicly and under what conditions.

  
**6\. Additional recommendations**

-   Review the repository before making it public
    
-   Use secret-scanning tools whenever possible
    
-   Keep the repository up to date with good documentation practices
    
-   When in doubt, opt to keep the repository private
