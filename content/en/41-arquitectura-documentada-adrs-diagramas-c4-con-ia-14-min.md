## Why diagrams and ADRs die first

Architecture documentation is the first to become outdated and the most costly not to have. The PowerPoint diagrams from the kick-off become obsolete in the first refactor sprint. The ADRs nobody writes turn into Slack conversations nobody can find two years later.

The problem isn't a lack of willingness — it's friction. Creating and maintaining a Lucidchart diagram carries too much cognitive cost to do it as part of a PR. Writing an ADR from scratch requires remembering the template, gathering the arguments, and dedicating 30-45 minutes to something that "isn't in the ticket".

The solution is to lower that friction to zero: **diagrams as text in the repo + ADRs generated by AI from the conversation you already had**.

## ADRs: the project's institutional memory

An **Architecture Decision Record (ADR)** is a short document that records a significant technical decision: the context in which it was made, the options considered, and the reason one was chosen over the others.

It's not about documenting every decision — only the ones a new developer would need to understand in order not to redo them or question them without context. The pragmatic criterion: *If this developer joined the project today and saw this code, would they ask "why did they do it this way"?* If the answer is yes, there's an ADR pending.

### The MADR format

The dominant format in 2025-2026 is **MADR** (Markdown Any Decision Records), not the original Nygard format. The key difference: MADR has explicit structure for the options considered and their pros/cons, which makes it much more useful when an AI agent has to fill it in.

```
# Use of SQLite + better-sqlite3 instead of PostgreSQL

## Status

Accepted

## Context and problem

The FlowSync project is a task management application with a predictable workload
and a team of 2-4 developers. We need a database for local development and for
the master's program testing environments.

## Considered options

* SQLite + better-sqlite3
* PostgreSQL in Docker
* Turso (distributed SQLite)

## Decision

**SQLite + better-sqlite3** is chosen because:

* Zero infrastructure overhead for local development
* AdonisJS 7 + Lucid ORM supports SQLite first-class
* better-sqlite3 is synchronous — it removes async complexity in simple queries
* For the real production of the master's project, the load doesn't justify Postgres

## Consequences

* The available column types are SQLite's (no arrays, no advanced jsonb)
* Migrating to Postgres in production requires reviewing the migrations (the types are compatible via Lucid)
* No support for concurrent write connections at scale — acceptable in this context
```

### Installation and setup with log4brains

**log4brains** is the CLI that combines MADR templates + static UI + GitHub Pages integration. It's the option with the best configuration-to-result ratio for JS/TS projects.

```
# Installation (once per project)
npm install --save-dev log4brains

# Initialization — generates docs/adr/ with the MADR template
npx log4brains init

# Create a new ADR
npx log4brains adr new "Uso de SQLite en lugar de PostgreSQL"

# Local preview of the ADR site
npx log4brains preview
```

The structure `log4brains init` creates:

```diagram
docs/
└── adr/
    ├── README.md              # Index of ADRs
    ├── 20260101-uso-de-adonisjs-7.md
    ├── 20260115-uso-de-sqlite.md
    └── template.md            # MADR template that npx log4brains adr new will use
```

> ⚠ **Naming convention**
> 
> `log4brains` names files with an ISO date + slug: `YYYYMMDD-nombre-del-adr.md`. Don't use manual sequential numbers — the date is more robust when there are concurrent ADRs on different branches.

### Generating an ADR with Claude Code

The most efficient flow is not writing the ADR — it's **transcribing the decision you already made** into MADR format. Claude Code does this very well if you give it the right context.

Example prompt in Claude Code:

```
Based on the following context, generate an ADR in MADR format for the project's
docs/adr/ directory.

Context: We decided to use VineJS for validation instead of Zod because AdonisJS 7
includes VineJS natively and it has direct integration with Lucid models.
Zod would work but introduces an additional dependency, and manually integrating
it with AdonisJS types requires more boilerplate. The team already knows the
VineJS syntax from working with the official AdonisJS v6 guide.

Name the file with today's date and the appropriate slug.
```

> 💡 **Productivity trick**
> 
> If the decision was made in a Slack thread or in a PR description, paste that text directly into the prompt. Claude Code extracts the arguments and structures them into MADR better than most developers would do writing it from scratch.

## Mermaid 11: diagrams that live in your Markdown

**Mermaid.js** is the standard solution for diagrams embedded in Markdown. GitHub, GitLab, and Notion render them natively — you write the diagram as text in a code block with the `mermaid` identifier and it appears as a visual.

Version 11 (current) adds dedicated architecture diagrams, Radar charts, and a customizable ELK layout engine. What matters most for an AdonisJS 7 project:

**Sequence diagram (authentication flow):**

```

    participant A as AdonisJS API
    participant DB as SQLite

    C->>A: POST /auth/login {email, password}

    DB-->>A: User record
    A->>A: Verify password hash (bcrypt)
    A->>DB: INSERT access_token (user_id, hash, expires_at)
    DB-->>A: Token record

    Note over C: Almacena token en localStorage / HttpOnly cookie
```

> **Note about Mermaid on GitHub**
> 
> It works well for most diagram types. Known limitations: clickable hyperlinks on nodes don't always work in GitHub Preview, and some FontAwesome symbols (`fa:fa-ban`) don't render. For production diagrams that need precision, use Structurizr DSL (see below).

The real advantage of combining Mermaid with Claude Code is that you can request diagrams **directly from the repo structure**, without having to describe the architecture manually.

## C4 with Structurizr DSL: the standard for real architectures

For more complex projects, or when you need multiple coordinated views (Context → Container → Component → Code), **Structurizr DSL** is the reference tool. It was created by Simon Brown, the author of the C4 model.

The central idea: you define the architecture model **once** in a `.dsl` file, and Structurizr generates all the views from that model. If you move a component, all the views update.

```
# flowsync.dsl — minimal C4 model of the project

workspace "FlowSync" "Sistema de gestión de tareas" {

    model {
        user = person "Usuario" "Developer o PM usando FlowSync"

        flowsync = softwareSystem "FlowSync" "Aplicación de gestión de tareas" {
            frontend = container "Frontend" "SPA React 19 + Vite" "TypeScript/React" {
                tags "Browser"
            }
            backend = container "Backend API" "AdonisJS 7 REST API" "TypeScript/Node 24" {
                authController = component "AuthController" "Login, registro, tokens"
                taskController = component "TaskController" "CRUD de tareas"
                userService = component "UserService" "Lógica de usuario"
            }
            database = container "Base de datos" "SQLite via better-sqlite3" "SQLite" {
                tags "Database"
            }
        }

        user -> frontend "Usa"
        frontend -> backend "API REST / JSON (Bearer token oat_*)"
        backend -> database "Lee y escribe vía Lucid ORM"
    }

    views {
        systemContext flowsync "Context" {
            include *
            autolayout lr
        }
        container flowsync "Containers" {
            include *
            autolayout lr
        }
        component backend "Components" {
            include *
            autolayout lr
        }
        theme default
    }
}
```

**To render locally:**

```
# With Docker (easiest way)
docker pull structurizr/lite
docker run -it --rm -p 8080:8080 \\
  -v $(pwd)/docs/architecture:/usr/local/structurizr \\
  structurizr/lite

# The .dsl file must be at docs/architecture/workspace.dsl
# Open <http://localhost:8080>
```

**To export SVG in CI** (GitHub Actions):

```
# .github/workflows/generate-diagrams.yml
- name: Generate C4 diagrams
  uses: structurizr/actions/export@main
  with:
    workspace: docs/architecture/workspace.dsl
    format: svg
    output: docs/diagrams/generated/
```

> 💡 **When to use Mermaid and when to use Structurizr**
> 
> Use **Mermaid** when the diagram is ad-hoc, lives in a README or ADR, and you don't need coordinated views. Use **Structurizr** when you have more than two layers of architecture, the team needs consistent views of the same model, or when architecture documentation is a formal deliverable. For the master's project, Mermaid in the README + a basic `workspace.dsl` in `docs/architecture/` is the right combination.

## Complete flow: Claude Code generates your architecture documentation

Here is the end-to-end flow you're going to run on the AdonisJS 7 project:

```
1. Claude Code reads app/models/*.ts
        ↓
2. Generates a Mermaid entity diagram (ERD or graph)
        ↓
3. Embeds it in README.md under ## Arquitectura
        ↓
4. Claude Code reads start/routes.ts + app/controllers/
        ↓
5. Generates a sequence diagram of the auth flow (Mermaid)
        ↓
6. Places it in docs/architecture/auth-flow.md
        ↓
7. You decide: does it deserve a Structurizr DSL for formal C4 views?
        ↓
8. If yes: Claude Code generates the base workspace.dsl
   If not: the Mermaid diagrams are enough for this phase
```

The "does it deserve Structurizr?" criterion is pragmatic: if the Mermaid diagram in the README explains the architecture well enough to a new developer, you don't need more. If you have multiple services or the system has more than 3-4 non-trivial containers, Structurizr pays for its complexity.
